# LLM Observability: UI Concept

## Overview

Each task proposal stores an `llm_call_profile` list inside `last_proposal.cli_result`. This profile is also forwarded to the acceptance runner's latency diagnostics. The frontend must distinguish **real** entries from **synthetic** ones before rendering any model-performance metrics.

## Entry Types

| `source` field | `estimated` field | Type | Meaning |
|---------------|-------------------|------|---------|
| `"model_invocation_service"` | `false` | **Real** | Actual LLM API call with measured latency and token counts |
| `"orchestrator_synthetic"` | `true` | **Synthetic** | Placeholder entry created by the orchestrator to record that a planning step occurred; no real call was made |
| any | `true` | **Estimated** | Token counts or latency are estimates, not measured values |

## Metric Rendering Rules

**Allowed in charts and dashboards:**
- `latency_ms` from real entries (`estimated=false`, `source != "orchestrator_synthetic"`)
- `prompt_tokens`, `completion_tokens` from real entries
- `success_count_real`, `fail_count_real` (keyed as `_real` in diagnostics output)
- `by_model_real` distribution

**Prohibited in charts:**
- Latency or token stats from synthetic entries — they are `null` and represent orchestration activity, not model performance
- Aggregated stats that mix real and synthetic without filtering
- `calls_seen_total` as a standalone KPI — always show `calls_seen_real` separately

**Warning rule:** If `calls_seen_synthetic > 0`, display a note: "X of Y calls are synthetic orchestration records and are excluded from performance statistics."

## Labels

| Field | Recommended display label |
|-------|--------------------------|
| `calls_seen_total` | "Total call records" |
| `calls_seen_real` | "Real LLM calls" |
| `calls_seen_synthetic` | "Synthetic/orchestration records" |
| `latency_ms_mean_real` | "Avg. latency (real calls, ms)" |
| `prompt_tokens_mean_real` | "Avg. prompt tokens (real calls)" |
| `fail_count_real` | "Failed real calls" |

## Warning Against Synthetic Metric Pollution

Synthetic entries are generated by the planning orchestrator to record that a planning strategy was attempted without invoking the LLM. They have `estimated=true` and `latency_ms=null`. Including them in latency histograms or token-cost calculations will produce misleading results:

- Mean latency will drop toward zero (null entries are typically excluded by averaging, but skew call-count-based KPIs).
- Token cost estimates will be understated if synthetic entries are counted as zero-token calls.
- "Calls per second" throughput metrics will be inflated.

**Always filter by `estimated=false` and `source != "orchestrator_synthetic"` before aggregating.**

## Provider Observer Integration

The provider-observer snapshot (`GET /api/system/provider-observer`) is separate from the LLM call profile. Use it for:
- Detecting whether a provider is currently reachable (pre-run health check).
- Classifying `provider_unavailable` idle time in latency diagnostics.

Do not conflate provider-observer status with LLM call success rates — the observer probes availability, not quality.
